  ---
  Diagnosis: 3 compounding causes of mean collapse

  Cause 1 — Cd loss weight is 0.1 in the config (dominant cause)

  In configs/f1_base.yaml:35:
  loss_weights:
    cp:  1.0
    wss: 1.0
    cd:  0.1   # ← this is the culprit

  Looking at epoch 30 from the training log, the actual gradient contributions
  are:
  - Cp: 0.0858 × 1.0 = 0.0858
  - WSS: 0.1191 × 1.0 = 0.1191
  - Cd: 0.0147 × 0.1 = 0.0015  ← 0.7% of total gradient

  The Cd signal is almost invisible. The model sees no meaningful pressure to
  spread its predictions — predicting the dataset mean (≈ 0.24) keeps Cd L1 loss
   at ~0.015 while costing nothing from the backbone's perspective.

  Cause 2 — y_cd is not normalized (data/drivaernet_dataset.py:171)

  cd = torch.tensor([raw['cd_total'] or 0.0], dtype=torch.float32)  # raw 
  ~0.20–0.29

  y_cp and y_wss are symlog + z-score normalized to std≈1.0. y_cd is stored raw.
   The Cd range is only 0.20–0.29 (physical std ≈ 0.02), so even a model with a
  properly-weighted gradient gets very little signal — all samples look nearly
  identical in Cd space.

  Cause 3 — Structural gradient imbalance

  Per-vertex losses (Cp: ~50k gradients/sample, WSS: ~50k×3) dominate the
  backbone. The global_mean_pool in GlobalHead collapses all spatial variation
  into a single mean vector — features shaped by Cp/WSS carry almost no
  discriminative information for global Cd drag.

  ---
  The scatter plot code is correct

  No bugs in your plotting script — it accurately reflects what the model is
  doing. y_cd is unnormalized and pred['cd'] is in the same raw space, so the
  comparison is apples-to-apples. The flat line at 0.24 is a faithful picture of
   the collapse.

  ---
  Fixes (easiest → most complete)

  Fix 1 — Increase Cd loss weight (5-minute fix, try first):

  In configs/f1_base.yaml:
  loss_weights:
    cp:  1.0
    wss: 1.0
    cd:  5.0   # was 0.1 — increase by 50×

  Fix 2 — Normalize y_cd in the dataset (data/drivaernet_dataset.py:171):

  # Dataset-wide Cd stats for DrivAerNet++ (compute once from your data)
  CD_MEAN = 0.24   # replace with actual mean from your dataset
  CD_STD  = 0.020  # replace with actual std from your dataset

  cd_raw = raw['cd_total'] or 0.0
  cd = torch.tensor([(cd_raw - CD_MEAN) / CD_STD], dtype=torch.float32)

  Store CD_MEAN and CD_STD in the Data object so you can denormalize at eval
  time. Your scatter plot code then needs one line: cd_true = cd_true * CD_STD +
   CD_MEAN to convert back to physical values.

  Fix 3 — Change Cd loss from L1 to MSE (fights mean collapse more
  aggressively):

  In configs/f1_base.yaml:
  loss_type:
    cd: "mse"   # MSE penalizes spread from mean more than L1 does

  Fix 4 (best, requires more work) — Physics-based Cd from Cp/WSS integration

  Instead of a separate GlobalHead learning Cd independently, compute it from
  the Cp and WSS predictions via a surface integral over the mesh. Then Cd
  automatically improves as Cp/WSS improve, and there's no gradient conflict at
  all. This is architecturally sound but requires storing face areas and outward
   normals in the graph data.

  Recommended order: Apply Fix 1 + Fix 2 + Fix 3 together and retrain. That
  should break the collapse. If Cd still doesn't improve after 50 epochs, Fix 4
  is the right long-term direction.

⏺ Good findings. Two things confirmed from the code:

  1. Vertex normals are computed (compute_vertex_normals in precompute_geometry)
   but only angles and transporters are stored in the Data object — normals are
  dropped after use
  2. Face areas are never stored — the cross product magnitudes are computed
  inside compute_vertex_normals only to do area-weighted averaging, then
  discarded

  So surface integration would require adding both face normals and face areas
  to the Data object. But you're right about the error propagation risk — let me
   reason through both issues properly before touching anything.

  On the weight: you're correct that cd: 5.0 will hurt Cp/WSS. Here's why 5.0
  was actually wrong even as a fix — before normalisation, Cd loss was tiny
  (~0.015) because targets were raw values in a narrow [0.20, 0.29] band. After
  Fix 2's z-score normalisation the Cd loss will be on the same scale as Cp/WSS
  (~0.5–1.0) from epoch 1. So w_cd = 1.0 is the right post-normalisation weight.
   No need for the 5× amplification.

  On physics-based Cd: your instinct is correct. The integration formula is:
  Cd = (1/A_ref) * Σ_faces [ Cp * q_inf * n_x + τ_x ] * A_face
  This would propagate Cp/WSS errors directly into Cd. Wherever the model
  mispredicts Cp at the tail, that error would be multiplied by the face area
  and accumulated. The current GlobalHead is actually more forgiving — it can
  learn to compensate for systematic field errors since it doesn't know where
  they are. So keep the GlobalHead; just fix the weight.

⏺ Fix 3: Global Context Injection — Detailed Explanation

  The problem in concrete terms

  Right now, the forward pass is purely local:

  Node p at the tail ──(hop 1)──> sees 1-ring neighbours
                    ──(hop 2)──> sees 2-ring neighbours
                    ...
                    ──(hop 5)──> sees 5-ring neighbours

  On a 50k-vertex car mesh, 5 hops covers roughly a 3–5 cm geodesic radius. The
  tail pressure is determined by what's happening at the front stagnation point
  (nose Cp ≈ +1.0), the roof (separated or attached flow), and the A-pillars.
  All of that is physically 1–2 m away. The model never sees it.

  ---
  The mechanism

  After each GEM block, you do two things:

  Step A — Pool: collapse all V node features into one vector representing the
  whole graph
  h_global = (1/V) * Σᵢ hᵢ        shape: (C,)

  Step B — Broadcast back + residual add: give every node a copy of that summary
  hᵢ ← hᵢ + W · h_global          for all i simultaneously

  W is a learned linear projection. The node's local information stays intact
  (it's a residual add, not a replacement). The global summary just gets added
  as an extra signal.

  After 5 blocks with this injection, a tail node has seen:
  - Block 1: its 1-hop local neighbours (local geometry)
  - Block 1 global: the mean state of the entire car after embedding
  - Block 2: its 2-hop local neighbourhood + carries the injected global context
   forward
  - Block 2 global: updated global state that now includes some processed
  features
  - ...and so on

  The tail node effectively "knows" the global aerodynamic state at every layer.

  ---
  Why global mean pool works here

  You might ask: doesn't averaging over all 50k vertices lose all spatial
  information? Yes — but that's exactly what you want. You're not trying to give
   the tail the location of the stagnation point. You're giving it a summary 
  statistic of the flow state:

  - "The average pressure-like feature across the car is X" → this tells the
  tail whether it's a high or low drag configuration
  - "The average vector-like feature pointing in Y direction" → encodes the
  dominant flow direction globally

  The tail doesn't need to know where things are globally. It just needs to know
   what kind of car this is globally so it can calibrate its local prediction
  accordingly.

  ---
  The concrete implementation in your code

  Currently in f1_net.py, the forward loop is:

  for block in self.blocks:
      h = block(h, edge_index, angles, transporters)   # (V, C_out)

  After the change:

  for block, global_proj in zip(self.blocks, self.global_projs):
      h = block(h, edge_index, angles, transporters)   # (V, C_out)  — local 
  update

      # Global context injection
      h_global = global_mean_pool(h, batch)            # (B, C_out)  — one 
  vector per graph
      h_global = global_proj(h_global)                 # (B, C_out)  — learned 
  projection
      h = h + h_global[batch]                          # (V, C_out)  — broadcast
   to all nodes

  h_global[batch] is the key line. batch is the vector that says which graph
  each node belongs to (all zeros for batch_size=1). This indexes the single
  global vector and broadcasts it to all V nodes — giving every node an
  identical copy of the global summary.

  In __init__, you add one nn.Linear per block:

  # One global projection per GEM block (after each block's output)
  self.global_projs = nn.ModuleList([
      nn.Linear(feature_dim(ftype_out), feature_dim(ftype_out), bias=False)
      for ftype_out in self.ftypes[1:]   # one per block output
  ])

  ---
  What flows through the graph after injection

  Here's a trace of what a tail node "knows" at each block, after injection:

  Block 1 output (before inject):  local 1-hop geometry features
  Block 1 inject:                 + mean of all nodes' 1-hop features (global
  shape summary)

  Block 2 input = Block 1 output + inject
  Block 2 output (before inject):  2-hop local features, BUT each node already
  carries
                                   global context from block 1 — so block 2's
  local
                                   aggregation is already "globally aware"
  Block 2 inject:                 + mean of all nodes' 2-hop + globally-aware
  features

  ...

  Block 5 input: already carries 4 rounds of global context.
                The tail node's features are influenced by the nose,
                roof, floor — every part of the car.

  This is why the injection at each layer is important, not just once at the
  start. At layer 1, the global summary is based on raw embedded features. By
  layer 3, the global summary reflects features that have already processed
  local geometry and previous global context — it's a progressively richer
  representation.

  ---
  The equivariance consideration

  One subtlety: your GEM-CNN is gauge-equivariant — features transform
  consistently under changes of local reference frame. Global mean pooling of
  the full feature vector (including ρ₁, ρ₂ components) would break this because
   you're averaging vectors in different local frames.

  The safe version: only pool the scalar (ρ₀) channels for the global summary.

  # Extract only ρ₀ (scalar) channels from h for pooling
  scalar_dim = sum(mult for order, mult in current_ftype_out if order == 0)
  h_scalars  = h[:, :scalar_dim]                      # (V, scalar_dim)
  h_global   = global_mean_pool(h_scalars, batch)     # (B, scalar_dim)  ← 
  gauge-safe
  h_global   = global_proj(h_global)                  # (B, scalar_dim)
  # Add only into the scalar channels
  h[:, :scalar_dim] = h[:, :scalar_dim] + h_global[batch]

  Scalars are gauge-invariant (they don't transform under frame rotation), so
  averaging them across vertices in different frames is mathematically correct.
  If you pool the full vector including ρ₁ channels, you're averaging arrows
  that point in different local directions — physically meaningless.
